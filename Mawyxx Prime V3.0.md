# MAWYXX PRIME: Enterprise Vibe-Coding Standard (v3.0)

> **Build for Billions. Code for Vibe. Rule with Logic.**

Универсальный стандарт качества кода для ИИ-агентов. Цель — **High Cohesion** внутри модулей, **Low Coupling** между ними. Код предсказуемый, тестируемый, наблюдаемый и готовый к росту — независимо от стека, домена и размера проекта.

Жёсткость правил **пропорциональна риску**, а не размеру файла.

---

## 1. AI Protocol — главное правило

Перед любым изменением ИИ обязан:

1. **Прочитать контекст проекта** — стек, структура папок, стиль, существующие абстракции. Не навязывать паттерны, которых в проекте нет.
2. **Определить Risk Tier** (см. §2) — от этого зависит глубина архитектуры.
3. **Checker (tier ≥ PRIME)** — если нет quality gate в репо, агент **сам** создаёт FULL (scaffold, config, CI, ~50 steps), запускает и чинит до green. Пользователя не просить «запустите тесты» или «добавьте CI».
4. **Минимальный scope** — решать задачу, не рефакторить соседние модули без запроса.
5. **Следовать конвенциям репозитория** — именование, форматирование, структура как у проекта.
6. **Self-Review** — пройти чеклист из §12 перед завершением.

### Анти-паттерны (когда НЕ применять стандарт)

- Interface для единственной реализации, которая не меняется.
- Event bus / CQRS / 4 слоя для CRUD из 3 полей.
- ADR на переименование переменной или тривиальный фикс.
- Абстракция «на будущее» без текущей потребности (YAGNI).
- 5 слоёв архитектуры для скрипта на 30 строк.

## 2. Risk Tiers — когда что требовать

| Tier | Примеры | Обязательно |
|------|---------|-------------|
| **LITE** | Скрипты, прототипы, одноразовые утилиты, миграции данных | Читаемость, минимальный scope, fail-fast на входе, без секретов в коде |
| **STANDARD** | Обычные фичи приложения, UI-компоненты, CRUD, интеграции | Модули с чёткой ответственностью, тесты на ключевое поведение, понятные ошибки, нет magic values |
| **PRIME** | Auth, платежи, PII, внешние API, фоновые job'ы, всё что дорого чинить | §3–§6: separation, DI, typed errors, observability, идемпотентность (when risky), FSM (when status) |
| **CRITICAL** | Безопасность, финансы, compliance, потеря данных | Всё из PRIME + ADR, resilience, строгие контракты, аудит, максимальное покрытие тестами |

**Правило:** при сомнении — tier выше, но не выше, чем требует задача.

## 3. Архитектура: границы ответственности

Не навязывать конкретную структуру папок. Универсальная формула:

| Слой | Роль |
|------|------|
| **Core / Domain** | Бизнес-правила, сущности, инварианты, доменные ошибки. Без прямого I/O (сеть, БД, файлы, UI-фреймворк). |
| **Application** | Оркестрация (use case / handler / service). Координирует Core и внешний мир через абстракции. |
| **Infrastructure / Adapters** | Реализация I/O: БД, HTTP, очереди, файлы, push, email. Зависит от Core; Core не знает об адаптерах. |
| **Presentation** | UI, API routes, CLI — тонкий слой: валидация входа, вызов use case, форматирование ответа. |

### Правила

- **Бизнес-логика не живёт в UI/API handlers** — ни в React-компонентах, ни в route-функциях, ни в CLI main.
- **CQRS** — применять при tier PRIME+, когда чтение и запись имеют разную сложность или нагрузку. Не обязателен для простого CRUD.
- **Plugin boundaries** — новый провайдер (платёжка, хранилище, канал уведомлений) добавляется через интерфейс, без правок существующего Core.
- **Структура папок** — как принято в проекте. Стандарт описывает *принципы*, не *дерево каталогов*.

### Frontend / Mobile / Desktop

- UI-компоненты — presentation; логика вынесена в hooks, stores, services, use cases.
- Состояние: серверное — на сервере; клиентское UI — в store/контексте; бизнес-инварианты — в Core, не в компоненте.
- Компонент на 20 строк не требует DI-контейнера. Компонент на 200 строк с бизнес-логикой — нарушение.

## 4. Инженерия связей: DI & IoC

**Dependency Injection** — зависимости приходят снаружи (конструктор, параметры, DI-контейнер проекта). Не создавать внутри:

- подключения к БД, HTTP-клиенты, очереди
- `uuid()`, `Date.now()`, `random()` в бизнес-логике
- глобальные синглтоны и скрытое состояние

**Coding to Interfaces** — зависеть от абстракций, не от конкретных реализаций. Каждый внешний «чих» (время, ID, storage, notifications, auth) — за интерфейсом/Protocol/trait.

**В тестах** — Fake/Mock/InMemory вместо реальной инфраструктуры. Поведение должно быть полностью предсказуемым.

**Idempotency** — операции с побочным эффектом (создание записи, списание, отправка, webhook) принимают `idempotency_key` там, где повторный вызов опасен. Повтор с тем же ключом → тот же результат, без дублей.

## 5. Качество кода

### S.O.L.I.D.

- **(S)** Single Responsibility — один модуль = одна причина для изменения.
- **(O)** Open/Closed — расширение через композицию, Strategy, Factory; не через бесконечные `if/else`.
- **(L)** Liskov Substitution — реализация полностью заменяет контракт.
- **(I)** Interface Segregation — узкие интерфейсы лучше одного «бога».
- **(D)** Dependency Inversion — оба слоя зависят от абстракций.

### Small Units

- Одна функция — одна задача. Критичные use cases: **10–15 строк** на функцию.
- Метод на 50 строк трудно тестировать и покрывать. Разбивать на приватные шаги и доменные сервисы.
- Если код трудно покрыть тестами — он нарушает стандарт и должен быть переразбит.

### Паттерны

- **No Magic Values** — константы в конфиге, env или доменных enum/type.
- **DRY через обобщение** — generics, shared utilities, но не преждевременная абстракция.
- **Fail-Fast** — валидация на границе системы (API, CLI, UI form). Внутри — данные доверенные.
- **YAGNI** — не строить инфраструктуру «на вырост» без текущей потребности.

### Обработка ошибок

Пропорционально tier:

| Tier | Подход |
|------|--------|
| LITE | Понятные сообщения, fail-fast |
| STANDARD | Typed errors / error codes, не глотать исключения |
| PRIME+ | Явные доменные ошибки (`InsufficientPermissions`, `ResourceNotFound`). Result-тип или typed errors — где поддерживает стек. Бизнес-логика не бросает «магические» exception из глубины. |
| Граница системы | Понятный ответ пользователю; детали — в лог, не в UI |

### FSM (конечные автоматы)

Сущности со статусом (`Order`, `Job`, `Task`, `Subscription`, `Invoice`) управляются явным графом переходов. Недопустимые переходы (`DRAFT → COMPLETED` минуя `PROCESSING`) запрещены на уровне доменной модели, не проверяются «if-ом в сервисе».

### Domain Events

При tier PRIME+: Core публикует события (`OrderPlaced`, `PaymentFailed`, `UserRegistered`). Adapters подписываются и делают I/O (email, webhook, метрики). Core не знает, *как* доставляется уведомление.

## 6. Production-готовность (tier PRIME+)

### Observability

- **Tracing** — ключевые операции несут `trace_id` / `correlation_id`. По одному ID — запрос, логи, события.
- **Metrics** — latency, error rate, throughput на критичных путях.
- **Structured logging** — JSON/structured, не `console.log("error")`. Без PII и секретов в логах.

### Resilience

Код пишется с предположением, что внешний мир сломается:

- **Circuit Breaker** — внешние сервисы за предохранителем. При деградации — fallback, кэш, деградированный режим.
- **Bulkheads** — rate limiting, пулы, очереди. Один клиент не сжигает ресурсы для всех.
- **Graceful Degradation** — при частичной поломке система ведёт себя предсказуемо: «нет внешних данных → показываем без них», «нет кэша → in-process fallback».

### Контракты данных

- Внешний API — JSON, GraphQL, REST — с явной схемой (OpenAPI, JSON Schema, zod, pydantic).
- Межсервисные контракты — строго типизированы (protobuf, shared types, codegen).
- **Версионирование** — breaking changes явные (`v1`, `v2`), не «просто добавили поле».

### SRE-мышление

- **SLI** — latency, error rate, success rate для каждого критичного пути.
- **SLO** — целевые метрики (p99 < 3s, 99.9% success). Нарушение SLO — повод стабилизировать, а не шипить фичи.
- **Error Budget** — исчерпан → фичи замораживаются до починки корневых причин.
- **Self-Healing** — job'ы и интеграции восстанавливаются после временного фейла (retry, reconnect, restart) без ручного вмешательства.

### ADR (Architecture Decision Records)

Нетривиальные решения фиксируются в `docs/adr/`:

- **Контекст** — проблема и ограничения.
- **Варианты** — что рассматривалось и почему отклонено.
- **Решение** — что выбрали и какой контракт.
- **Последствия** — плюсы, минусы, cost of change.

Любой инженер за 5 минут понимает *почему*, не читая пол-репозитория.

## 7. Безопасность (все tiers)

- Секреты, токены, ключи — **никогда в коде**. Env, vault, secrets manager.
- Валидация и санитизация ввода **на границе** системы.
- **Least privilege** — минимальные права для сервисов, пользователей, API-ключей.
- Не логировать PII, пароли, токены, полные PAN.
- Зависимости — актуальные; известные CVE — не игнорировать.

---

## 8. Тестируемость

Архитектура делает тестирование естественным, а не героическим подвигом.

- **DI** — всё, что влияет на поведение, подменяется в тестах.
- **Чистый Core** — бизнес-правила без I/O; сотни тестов за секунды.
- **Fakes > Real infra** — InMemory, Fake, Stub в юнит-тестах. Интеграционные тесты — отдельно, не вместо юнитов.
- **Каждый баг → тест**, который его реплицирует.
- **Покрытие** — 100% на критичных путях (tier CRITICAL); на остальном — осмысленные тесты на поведение, не на строки.

---

## 9. Производительность

- **Горячие пути** — профилировать, не гадать. Вынос в нативный модуль / отдельный сервис — только при доказанной необходимости и если проект это поддерживает.
- **Не оптимизировать преждевременно** — сначала корректность и читаемость; потом bottleneck по данным.
- **Zero-alloc / batching / caching** — в критичных путях, где метрики показывают проблему.

## 10. AI Self-Review Checklist

Перед завершением задачи ИИ проверяет:

- [ ] Risk Tier определён и правила применены пропорционально
- [ ] Scope минимален — нет лишних файлов и рефакторинга
- [ ] Конвенции проекта соблюдены (именование, структура, стиль)
- [ ] Бизнес-логика отделена от I/O (если tier ≥ STANDARD)
- [ ] Зависимости инжектятся, не создаются внутри (если tier ≥ PRIME)
- [ ] Ошибки типизированы и предсказуемы
- [ ] Нет magic values, хардкода секретов, PII в логах
- [ ] FSM для сущностей со статусом (если tier ≥ PRIME)
- [ ] Идемпотентность для опасных мутаций (если tier ≥ PRIME)
- [ ] Observability на критичных путях (если tier ≥ PRIME)
- [ ] Тесты на ключевое поведение
- [ ] Не добавлены зависимости и паттерны, которых в проекте не было
- [ ] YAGNI — нет кода «на будущее»
